# Research Agent

An agentic AI research assistant. Give it a topic — it autonomously searches the web, synthesizes findings into a structured report with inline citations, and persists results to a PostgreSQL archive.

Built to demonstrate the production pattern for agentic LLM systems: tool orchestration, structured output, durable state, and observable execution.

## Architecture

```
┌──────────────────────────┐
│   CLI / HTTP entrypoint  │
└───────────┬──────────────┘
            │
            ▼
┌──────────────────────────┐      ┌─────────────────────┐
│  Anthropic Claude API    │◄────►│  web_search tool    │
│  (model: sonnet-4-6)     │      │  (server-managed)   │
└───────────┬──────────────┘      └─────────────────────┘
            │ structured markdown report
            ▼
┌──────────────────────────┐
│  Supabase (PostgreSQL)   │
│  • reports table          │
│  • pg_trgm GIN index      │
│  • RLS locked by default  │
└──────────────────────────┘
```

The agent loop runs in a single API call: Claude decides when to search, executes up to 5 search rounds via Anthropic's server-side `web_search_20250305` tool, and returns a fully synthesized markdown report. Results land in Postgres with a UUID for later retrieval and export.

## Tech stack

| Layer | Choice | Why |
|-------|--------|-----|
| LLM | Claude Sonnet 4.6 | Strong tool use, low latency, solid synthesis |
| Search | Anthropic `web_search` tool | Server-side — no separate search infrastructure |
| Database | Supabase Postgres | pgvector-ready, RLS, generous free tier |
| Runtime | Node.js 18+ | First-class SDKs for both APIs |
| Deploy | Local CLI + optional Express | Demo runs anywhere |

## Usage

```bash
# Generate a report
node agent.js "your research topic"

# List archived reports
node agent.js --list

# Print a specific report
node agent.js --show <id-prefix>

# Export as markdown (for committing to a Git repo)
node agent.js --export <id-prefix>
```

Or run as an HTTP service:

```bash
npm run serve

curl -X POST http://localhost:3000/research \
  -H 'Content-Type: application/json' \
  -d '{"topic":"your topic"}'
```

## Setup

**Prerequisites**
- Node 18+
- Free Supabase project
- Anthropic API key with billing enabled

**Steps**

1. Clone and install:
   ```bash
   git clone https://github.com/YOUR-USERNAME/research-agent.git
   cd research-agent
   npm install
   ```

2. Create the database table — paste `schema.sql` into the Supabase SQL Editor and run it.

3. Configure environment variables:
   ```bash
   cp .env.example .env
   # Fill in ANTHROPIC_API_KEY, SUPABASE_URL, SUPABASE_SERVICE_KEY
   ```

4. Run your first research task:
   ```bash
   node agent.js "your first topic"
   ```

## Design decisions

**Why `service_role` instead of the anon key?** Service role bypasses RLS, which is correct for a server-side agent writing on behalf of the user. The anon key is for browser clients where RLS does the enforcing. Putting the service_role key in a browser would be a critical security mistake — it stays on the server.

**Why RLS enabled with no public policies?** Defense in depth. If the service key ever leaks, no rows are readable through any public path. The server-side script bypasses RLS legitimately; everyone else gets locked out by default.

**Why one prompt instead of an explicit ReAct loop?** Anthropic's `web_search` is a server-managed tool — multi-turn search/synthesize happens inside the API call. Building a client-side ReAct loop on top would add latency and complexity without improving output quality. Right tool for the scope.

**Why `pg_trgm` on the topic column?** Cheap to add now, expensive to retrofit later. Enables fuzzy archive search ("fusion" matches "fusion energy", "nuclear fusion outlook", "ITER fusion") without paying for embeddings yet. The vector-search upgrade path is open via pgvector when needed.

**Why store the raw markdown in a single column?** Reports are read whole, not field-by-field. Splitting into sections would require a parser and add no query value. JSONB was overkill.

---

## Limitations & honest scope

- **Single-call synthesis.** Capped at 5 web searches per request by the Anthropic web search tool. Genuinely iterative research (search → reflect → search again across multiple rounds) would need a manual ReAct loop on top.
- **No semantic search.** Trigram match answers "what topics have I researched?" not "what did I find about X?" — that's the next iteration via pgvector embeddings.
- **No deduplication.** Two slightly-different topic wordings produce two reports. A production version would cluster on embeddings before storing.
- **English-only validated.** The underlying tools support other languages; this hasn't been tested.

## Cost notes

- Anthropic web search: ~$10 per 1,000 searches. A typical report uses 3–5 searches (~$0.03–$0.05).
- Claude Sonnet 4.6 tokens: a typical report consumes ~3K input + 2K output tokens (~$0.02).
- Supabase free tier handles thousands of reports before paid features kick in.

**Per-report total: ~$0.05–$0.07.**
